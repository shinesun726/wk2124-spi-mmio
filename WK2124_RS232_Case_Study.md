# 通过绕开标准 SPI 框架解决 RK3568 + WK2124 高负载 RS232 丢包问题

## 1. 案例概述

这是一次典型的“现象像线程调度问题，但根因其实在访问路径确定性不足”的问题排查案例。

项目平台使用 RK3568，外接 WK2124 SPI 转多串口芯片（硬件FIFO 256B），对上层仍以标准串口节点的形式提供服务。在客户现场，系统需要同时运行多个 RT 优先级进程，并承载较大的串口吞吐量。在这种高负载场景下，RS232 通信出现了丢数据问题。

这次问题的难点不在于“有没有办法改善”，而在于“如何彻底解决”。我们尝试过提高优先级、绑核、RT 内核、DMA 等常规手段，现象都有不同程度改善，但都没有做到彻底稳定。最终真正有效的方案，是绕开传统 Linux SPI 框架的数据传输路径，直接通过 MMIO 方式访问 RK3568 的 SPI3 控制器寄存器，同时保留 WK2124 驱动接入 tty 子系统的能力。

从当前验证结果看，这样修改后问题有效消失，未再出现丢数据。

## 2. 项目背景

### 2.1 系统结构

- SoC 平台：RK3568
- 串口扩展芯片：WK2124
- 接口关系：RK3568 通过 SPI3 与 WK2124 通信
- 对外接口：上层仍通过标准串口节点访问，例如 `/dev/ttyWKx`

相关硬件描述可见：

- SPI3 控制器节点：[rk3568.dtsi](rk3568.dtsi#L3045)
- 板级 WK2124 节点：[rk3568-spring2.dtsi](rk3568-spring2.dtsi#L375)

其中，板级配置中 WK2124 挂在 SPI3 的片选 0 上，`spi-max-frequency` 为 10 MHz，并通过 `irq-gpio` 上报中断。

### 2.2 业务场景特点

本项目不是普通低负载串口场景，而是同时具备以下特点：

- 系统中存在多个 RT 优先级进程
- 串口吞吐数据量较大
- 对数据完整性要求高，包含诊断信息，不能接受偶发丢包
- 仍希望保留标准 tty 使用方式，而不是改成专用用户态服务

也正因为这几个条件叠加，问题并没有停留在一般驱动调参层面，而暴露为一个“系统调度时延和 I/O 路径确定性”的综合问题。

## 3. 问题现象

在高负载运行过程中，WK2124 对应的 RS232 串口会发生丢数据。该问题并非每次必现，但在客户实际业务压力下会稳定暴露。

表面上看，这像是以下几类问题之一：

- 驱动线程优先级不够，导致处理不及时
- 中断触发阈值不合适，导致读取不及时
- CPU 被高优先级任务抢占，SPI 访问延后
- SPI 频率、DMA 或数据拷贝路径存在瓶颈

但后续排查表明，这些都只是表层切入点，并不是最终根因。

## 4. 排查过程

### 4.1 第一阶段：怀疑是调度优先级问题

最先考虑的是驱动处理线程在高负载下得不到及时调度，因此首先尝试提升相关任务的实时性。

#### 尝试 1：改变任务优先级，把 SPI3内核线程 从普通优先级改为 RT 优先级

测试结果：丢数据大幅改善。

结论：

- 问题与系统调度延迟强相关
- 但“改善”不代表“解决”，说明优先级不是唯一决定因素

#### 尝试 2：提高 WK2XXX 线程优先级到 `rt99`

测试条件：普通内核。

结论：

- 该方向合理，因为 WK2124 的中断处理和数据搬运确实依赖线程及时运行
- 但这类改法本质上仍是在现有 SPI 框架路径上争抢调度资源，只能降低概率，不能消除路径上的不确定性

### 4.2 第二阶段：怀疑是 CPU 抢占导致 SPI 线程服务不及时

#### 尝试 3：隔离 `cpu3`，只负责调度运行 `spi3` 这个内核线程

测试结果：20M 数据量下，仍发生 4 次丢数据。

结论：

- 绑核说明问题和 CPU 抢占有关
- 但即使给 SPI 线程创造更好的 CPU 条件，仍然不能彻底避免丢包
- 这意味着瓶颈不仅是“线程有没有跑起来”，还包括“跑起来之后，这条访问路径是否足够短、足够确定”

### 4.3 第三阶段：尝试更重的系统级方案

#### 尝试 4：应用 RT 实时内核补丁

测试条件：未优化 `rkbin`，未隔离核心。

测试结果：

- 700M 数据量下，仍发生 4 次丢数据
- 实时性能提升约 30%

进一步情况：

- 若继续优化 `rkbin` 并隔离 `cpu3` 作为实时核心，该组合方案当时未完成测试闭环
- 实际推进过程中还发现 RT 补丁方案会带来系统卡死问题，稳定性不足

结论：

- RT 内核说明“调度时延”确实是重要因素，因为实时性能有明显提升
- 但即使如此，依旧存在小概率丢数据，说明 RT 内核只能缓解问题，不能彻底解决问题
- 更关键的是，RT 内核还引入了系统稳定性风险，因此不适合作为最终交付方案

### 4.4 其他尝试

除了上述重点路径外，还做过以下方向的尝试：

- 调整 WK2124 中断触发阈值
- 使用 DMA 传输
- 优化驱动中的数据拷贝
- 降低 SPI 时钟频率

这些手段都有一定工程合理性，但都没有从根本上解决问题。原因在于，它们仍然默认沿用传统 Linux SPI 框架的数据传输路径，而真正的问题恰恰就出在这条路径本身的实时确定性不足。

## 5. 为什么 RT 内核不是最终答案

这部分是整个案例里很值得讲的一点。

很多人在遇到“高负载 + 驱动线程响应不及时”的问题时，第一反应就是切 RT 内核。这次项目里，这条路我们也认真验证过，但最后明确放弃，原因有两个。

### 5.1 它能改善，但不能闭环

RT 内核带来了约 30% 的实时性能提升，说明方向并非完全错误。但在 700M 测试下仍然发生 4 次丢数据，问题并没有被彻底消除。

这说明：

- 调度只是问题的一部分
- 真正的关键在于 SPI 访问路径整体仍然存在不可控时延

### 5.2 它还引入了新的系统风险

实际应用 RT 补丁后，系统运行出现卡死问题。这一点非常关键，因为一个只能“部分改善通信问题”、却会破坏系统整体稳定性的方案，不可能成为最终交付方案。

因此，这次排查的关键判断不是“RT 内核没用”，而是：

> RT 内核证明了问题和调度有关，但它没有把问题解决到根上。真正要处理的，是 SPI 控制器访问链路本身的确定性。

## 6. 真实根因

### 6.1 表象根因

WK2124 在接收数据时，如果驱动侧不能及时通过 SPI 把数据读走，FIFO 就可能来不及被排空，进而触发溢出或形成丢数据现象。

### 6.2 真实根因

真实根因不是单纯的“WK2124 线程优先级不够”，而是：

> 在高负载场景下，传统 Linux SPI 框架的数据传输路径引入了不可控的调度延迟和额外的软件栈开销，导致 WK2124 的数据读取窗口被错失，最终在大吞吐串口场景下出现小概率甚至持续性的丢数据。

也就是说，问题不是只发生在“中断到了以后线程没被调度”；更深一层的问题是：

- 即使线程被调度了
- 即使优先级提高了
- 即使绑核了

只要 SPI 数据收发仍然经过传统框架路径，就仍然存在不可控的软件调度和处理延迟。

这就是为什么很多看起来合理的手段只能改善，不能闭环。

## 7. 最终解决方案

### 7.1 思路来源

在继续硬调内核线程优先级之前，我们先验证了一个更激进的方向：直接访问 RK3568 SPI 控制器寄存器。

用户态验证参考了仓库中的 [rkspi.c](rkspi.c)。这个方案通过 `ioremap` 方式直接操作 SPI 控制器寄存器，绕开了传统 SPI 驱动框架。验证结果表明，这样可以彻底解决丢数据问题。

但是，纯用户态直访寄存器还有一个明显缺点：

- 无法继续直接使用标准 `/dev/ttyWKx` 节点
- 往往需要额外设计专用 server 进程承接串口通信
- 工程集成成本高，业务使用方式也会改变

因此，最终方案不是停留在用户态验证，而是把这个思路迁回内核驱动。

### 7.2 方案落地方式

最终采用的方式是：

- 保留 WK2124 驱动接入 tty 子系统的整体结构
- 不再依赖传统 SPI 框架提供的数据传输接口作为关键收发路径
- 直接映射 RK3568 SPI3 控制器寄存器
- 通过 MMIO 方式直接轮询 TX/RX FIFO 完成数据收发

这样做的结果是：

- 上层仍然可以继续按标准串口方式使用设备
- 底层关键路径从“通用 SPI 框架路径”变成了“短路径、可控路径”
- 大幅降低了高负载下的访问不确定性

## 8. 关键代码证据

下面这些代码，是支撑这次结论的核心证据。

### 8.1 MMIO 抽象结构

在 [spi-wk2xxx.c](spi-wk2xxx.c#L329) 中定义了 `struct wk2xxx_mmio_spi`，其中保存了：

- SPI 控制器寄存器映射地址
- FIFO 长度
- FIFO 读阈值
- SPI 时钟分频信息
- 片选信息

这说明驱动已经不再只是“调用标准 SPI 传输接口”，而是在驱动内部持有控制器级别的直接访问能力。

### 8.2 控制器初始化

[spi-wk2xxx.c](spi-wk2xxx.c#L696) 中的 `wk2xxx_mmio_init()` 完成了几件关键事情：

- 从父设备节点获取 SPI 控制器资源
- 通过 `devm_ioremap()` 映射 SPI3 控制器寄存器
- 获取 `spiclk` 和 `apb_pclk`
- 计算时钟分频
- 获取 FIFO 深度并建立 MMIO 传输参数

这部分直接对应了硬件基础信息来源：

- [rk3568.dtsi](rk3568.dtsi#L3045)
- [rk3568-spring2.dtsi](rk3568-spring2.dtsi#L375)

### 8.3 核心传输函数

[spi-wk2xxx.c](spi-wk2xxx.c#L597) 中的 `wk2xxx_mmio_xfer()` 是整个方案最关键的实现。

它的核心特点是：

- 直接读写 SPI 控制器寄存器
- 直接监控 `txflr`、`rxflr`、`sr` 等状态寄存器
- 在控制器 FIFO 层面主动决定何时写入、何时读取
- 通过 `cpu_relax()` 和超时控制维持短路径轮询

这与标准 SPI 框架最大的不同在于：

- 不再依赖通用框架调度一层层把传输请求下发
- 不再把关键延迟交给通用软件栈吸收
- 关键路径完全收拢到驱动自己的可控逻辑中

### 8.4 中断与数据处理路径

中断线程和端口处理路径仍然保留在驱动中：

- `wk2xxx_rx_chars()`：[spi-wk2xxx.c](spi-wk2xxx.c#L1001)
- `wk2xxx_tx_chars()`：[spi-wk2xxx.c](spi-wk2xxx.c#L1122)
- `wk2xxx_port_irq()`：[spi-wk2xxx.c](spi-wk2xxx.c#L1253)
- `wk2xxx_ist()`：[spi-wk2xxx.c](spi-wk2xxx.c#L1316)

这说明最终方案不是“另起炉灶重写整个串口栈”，而是保留原有 tty 接入框架，只替换最关键的 SPI 数据收发部分。

这也是工程上最有价值的一点：

- 既保住了现有接口兼容性
- 又真正命中了性能瓶颈

## 9. 最终结果

结合已有测试与当前验证，结论可以概括为：

- 提高任务优先级到 RT：丢数据大幅改善，但不是闭环方案
- 隔离 `cpu3` 给 `spi3` 内核线程：20M 下仍有 4 次丢数据
- 切换 RT 实时内核：700M 下仍有 4 次丢数据，且系统存在卡死风险
- 最终 MMIO 改造方案：目前看来有效，且没有再发生丢数据问题

如果后续需要对外做更正式的技术报告，建议继续补充两类量化信息：

- 最终方案下的连续运行时长
- 最终方案下的总数据量与丢包统计

但即使按现有证据，也已经足够得出清晰结论：

> 真正有效的不是“把线程提得更高”，而是“把关键路径缩得更短”。

## 10. 这次案例中我的关键贡献

如果把这次事情作为个人案例来讲，我认为可以把个人贡献总结为以下几个点。

### 10.1 没停留在表面优化

面对丢数据问题，没有把“优先级调高后现象改善”误判成问题已经解决，而是继续追问为什么仍有残余丢包。

### 10.2 正确识别了优化层级

通过 RT 优先级、绑核、RT 内核等多种尝试，逐步确认问题和调度有关，但最终进一步判断出：

- 调度只是现象触发条件
- 标准 SPI 框架路径的不确定性才是关键瓶颈

### 10.3 用用户态实验先做方向验证

在重改内核驱动之前，先通过寄存器直访方案验证技术方向是否成立，避免一开始就在内核态做高成本试错。

### 10.4 做到了“既解决问题，又保留原有接口形态”

最终方案并不是为了性能而牺牲工程可用性，而是在保留 tty 接口兼容性的前提下，完成底层关键路径优化。

这类方案比单纯“写一个能跑的新程序”更有工程价值，因为它兼顾了：

- 稳定性
- 兼容性
- 可交付性

## 11. 这次案例的可复用经验

这次问题排查有几个经验可以迁移到其他项目。

### 11.1 改善不等于解决

优先级提高、绑核、RT 内核都能改善现象，但如果问题仍存在小概率发生，就说明根因没有被命中。

### 11.2 不要默认通用框架一定适合极端实时场景

Linux 通用框架的价值是通用性、抽象性和可维护性，但在高吞吐、低时延、强确定性场景下，通用框架并不一定是最优解。

### 11.3 先做低成本方向验证，再做高成本系统改造

先在用户态验证寄存器直访的方向是否有效，再把方案迁回内核，这是一个很实用的工程方法。

### 11.4 真正好的方案，应该同时满足三件事

- 能解释根因
- 能稳定解决问题
- 能被系统工程接受

## 12. 面试讲述版

下面这段内容可以直接作为面试里的 2 到 3 分钟讲述稿。

我之前做过一个 RK3568 平台上的串口疑难问题排查。系统里用了 WK2124 这个 SPI 转串口芯片，对上层还是以标准 tty 节点方式提供串口服务。客户场景里会同时跑多个 RT 优先级进程，串口吞吐量也比较大，在这种高负载下出现了 RS232 丢数据。

一开始我们按常规思路做过很多尝试，比如提高任务优先级到 RT、把 wk2xxx 线程提到 rt99、隔离 cpu 给 spi3 线程、上 RT 实时内核、调中断阈值、尝试 DMA、优化拷贝和降 SPI 时钟。其中一些手段确实能改善，比如提 RT 优先级后丢包大幅改善，RT 内核实时性能也提升了大约 30%，但这些方案都没有彻底解决问题，RT 内核甚至还带来了系统卡死风险。

后来我判断问题不只是线程优先级，而是传统 Linux SPI 框架路径在高负载下存在不可控时延，导致 WK2124 的 FIFO 不能被及时读空。为了验证这个判断，我先用用户态直接访问 RK3568 SPI 控制器寄存器的方式做实验，结果发现丢包问题可以彻底消失。

基于这个验证，我再把思路迁回内核驱动里，保留原来接入 tty 子系统的框架，只把关键 SPI 收发路径改成 MMIO 直接访问控制器寄存器、主动轮询 FIFO。这样既保留了 `/dev/ttyWKx` 的使用方式，又绕开了标准 SPI 框架的关键时延。最终这个方案有效解决了丢数据问题，而且到目前为止没有再复现。

这个项目对我最大的提升，是让我更清楚地区分“现象改善”和“根因解决”，以及在系统性能问题上，应该优先找真正的关键路径，而不是只在参数层面不断调优。

## 13. 一句话总结

这次问题的本质，不是简单的线程优先级不够，而是在高负载条件下，标准 SPI 框架路径的实时确定性不足。最终通过 MMIO 直访寄存器缩短关键路径，才真正把问题解决到根上。
